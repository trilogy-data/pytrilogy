# Plan: rowset-output collapse bridge gap (shared by `merge` + scoped `join`)

**Status:** PLAN (no code changes yet). Supersedes the diagnosis in
`bug_scoped_join_collapses_same_base_and_dim_expansion.md`, whose central claim
("the model-merge form resolves; only the scoped-join form fails") is **FALSE**
‚Äî see Finding 1.

## Confirmed findings (all reproduced on `no_more_multiselect`, sf=0.01 models)

Discriminator probes via `engine.generate_sql(text)[-1]` against the
`tpc_ds_duckdb` models:

| Case | `merge` form | `join` form |
|---|---|---|
| plain FK ‚Üí dim ROOT key (no rowset), e.g. `physical_sales.customer.id` ‚Üí `customer.id` | ‚úÖ resolves | ‚úÖ resolves |
| **rowset output ‚Üí dim ROOT key** (Bug A: `bought.physical_sales.customer.id` ‚Üí `customer.id`) | ‚ùå "not connected" | ‚ùå "not connected" |
| rowset output ‚Üí rowset output (two-rowset workaround, shipped q46/q68) | ‚úÖ resolves | ‚úÖ resolves |
| same-base rowset ‚Üî rowset self-join (Bug B: `curr.brand_id` ‚Üî `prev.brand_id`) | n/a | ‚ùå predicate erased / output col dropped |

**Finding 1 ‚Äî it is NOT a join bug.** A plain global
`merge bought.physical_sales.customer.id into ~customer.id;` fails with the
identical `UnresolvableQueryException` in *both* directions. The scoped `join`
reuses `merge`'s collapse machinery (`_build_scoped_merge_index` ‚Üí
`Factory._build_concept` substitution), so it inherits the gap.

**Finding 2 ‚Äî fix target is v3 (the default engine).** v3 raises "Could not
resolve connections". v4 fails too, but it is explicitly **out of scope** here ‚Äî
all work below targets v3 + the shared collapse machinery.

**Finding 3 ‚Äî the asymmetry (why rowset‚Üírowset works, rowset‚ÜíROOT doesn't).**
After the collapse of rowset output `X` onto target `Y`:
- `env.concepts[X]` is *replaced* by `Y` (the target concept), and `X` survives
  only in `env.alias_origin_lookup[X]` as a ROWSET concept with pseudonym `{Y}`.
- The rowset's `derived_concepts` still lists `X`; the rowset's *other* output
  grains are rewritten to reference `Y` directly (e.g. `bought.bought_city`'s
  grain becomes `Grain<bought.bought_city, ‚Ä¶, customer.id>`).
- When `Y` is itself **ROWSET-produced** (rowset‚Üírowset), `Y` is generated by a
  rowset boundary node, so both rowsets produce `Y` ‚Üí the FINAL merge joins them.
- When `Y` is **ROOT** (datasource-backed), nothing re-registers the *rowset* as
  a producer of `Y`. The rowset boundary exposes `X` (a ROWSET pseudonym of `Y`),
  but discovery won't use that pseudonym FK as a join key into `Y`'s datasource.
  The rowset's measures and the dimension end up in disconnected graph
  components joined only through the rowset node, which never offers the FK as a
  join key.

**Finding 4 ‚Äî Bug B is a second facet.** `curr.brand_id` and `prev.brand_id`
both trace to base `deduped.sales.item.brand_id`. The union-find collapse maps
`curr.brand_id ‚Üí prev.brand_id`, fusing two *distinct filtered row sets* into one
concept, so the equi-join `curr.brand_id = prev.brand_id` degenerates and the
self-join predicate vanishes (now: 3 keys survive, `brand_id` predicate + output
column silently dropped ‚Äî same root, post-#577 manifestation).

## Root cause (one sentence)

The collapse machinery models a join as **concept identity** (replace source
with target). That is correct only for genuine same-entity ROOT=ROOT keys; for a
rowset-derived key it severs the rowset from the target's datasource (Bug A), and
for two same-base rowset keys it erases the real self-join predicate (Bug B).

## Fix design

Two facets, ideally one discriminated mechanism:

**Facet A (Bug A) ‚Äî bridge, don't sever.** When a join/merge key is rowset- (or
otherwise non-datasource-) derived, discovery must be able to use it as a *join
key* to the other side's concept instead of collapsing it away ‚Äî i.e. the rowset
boundary's pseudonym-carrying output must connect to the target datasource for
enrichment.

**Facet B (Bug B) ‚Äî keep distinct + emit predicate.** When both keys trace to
the same base but belong to distinct filtered rowsets (self-join), do not fuse
them into one concept; keep them distinct and emit a real equi-join predicate.

**Recommended unifying approach:** discriminate inside the collapse so it only
*fully collapses* genuine same-entity **ROOT=ROOT** keys (preserving today's
merge-like behaviour, incl. `test_three_way_join_collapses_like_merge`). For any
join where **either side is rowset/derived, or both keys share a base lineage**,
register a **query-scoped join edge** (a join path in the discovery graph) rather
than a concept-identity collapse. This is net-new: there is currently no
query-scoped join-edge mechanism ‚Äî the join *is* the collapse.

## Discriminator details to confirm during implementation

- "same base lineage" detection: walk both keys' `RowsetItem` content chains to
  their root base address; if equal ‚Üí self-join, keep distinct.
- "either side rowset/derived": check `derivation == Derivation.ROWSET` (or more
  generally not directly datasource-bound) on either key's built concept.
- ROOT=ROOT genuine-merge case must stay on the collapse path unchanged.
